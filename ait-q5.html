1 HTTP Response Headers

1.1 The important ones are:

  • Allow
    • Specifies what methods (i.e. GET, POST, PUT, DELETE, etc) are allowed for a given resource.
  • Content-Type
    • The MIME type of the content, informs the browser about how to render the material
    • A media type (formerly known as MIME type) is a two-part identifier for file formats and format contents transmitted on the Internet.
      • Examples:
        • application/{javascript,json,xml,pdf}
        • text/{css,html,xml,csv}
        • image/{png,jpeg}
  • Date
    • Date and time of message in format: Date: DOW, dd, mon, yyyy hr:min:sec tz
  • Expires
    • timestamp after which the response is deemed no longer valid by the server (same format as date)
  • Set-Cookie
    • instructions for cookies to store
    • EX: Set-Cookie: UserID=JohnDoe; Max-Age=3600; Version=1

2 Elementary Site Security

2.1 XSS

  • Cross-site scripting
  • exploits the trust a user has for a particular site
  • In a cross-site scripting attack, the attacker makes you involuntarily execute client-side code, most likely Javascript. A typical reflected XSS attacking attempt could look like this:
    https://security.stackexchange.com/search?q="><script>alert(document.cookie)</script>
  • Client-side attack
  • Comes from the SAME domain that the user is purposefully using. Exploits poor input sanitization on the part of the site developer.

2.2 Session Fixation

  • When a user logs onto a site, the site assigns them a "session id" so it knows not to ask them for a password any time they access a different page on that "session." But some websites encode their session ids in "weak" ways. For example, they could implement it so that it DOESN'T insist upon some predetermined format. So if someone else had access to your session id, or even worse, someone PROVIDED you a session id that they already know, and you unwittingly use it, and enter your password, then that same person can use your session id to get into the platform under the authentication YOU provided.
  • Here's an example from Wikipedia that sums it up really well:
    1. Mallory has determined that http://unsafe.example.com/ accepts any session identifier, accepts session identifiers from query strings and has no security validation. http://unsafe.example.com/ is thus not secure.
    2. Mallory sends Alice an e-mail: "Hey, check this out, there is a cool new account summary feature on our bank, http://unsafe.example.com/?SID=I_WILL_KNOW_THE_SID". Mallory is trying to fixate the SID to IWILLKNOWTHESID.
    3. Alice is interested and visits http://unsafe.example.com/?SID=I_WILL_KNOW_THE_SID. The usual log-on screen pops up, and Alice logs on.
    4. Mallory visits http://unsafe.example.com/?SID=I_WILL_KNOW_THE_SID and now has unlimited access to Alice's account.

2.3 CSRF

  • cross-site request forgery
  • exploits the trust that a site has in a user's browser
  • CSRF commonly has the following characteristics:
    • It involves sites that rely on a user's identity.
    • It exploits the site's trust in that identity.
    • It tricks the user's browser into sending HTTP requests to a target site.
    • It involves HTTP requests that have side effects.

2.4 Third-party tracking cookies

  • sourced from ANOTHER domain
  • runs javascript that collects information from YOUR cookies
  • allowed for the sake of advertisements, but can be disabled
  • first-party cookies are important and useful!
    • Helps the site know that you're you while you're using the site so it can provide you an experience consistent with your use of the site
  • third-party cookies are un-trusted because you signed up to go to a.com (first party), but b.com (third party) wants your information.

2.5 Replay attack

Suppose Alice wants to prove her identity to Bob. Bob requests her password as proof of identity, which Alice dutifully provides (possibly after some transformation like a hash function); meanwhile, Eve is eavesdropping on the conversation and keeps the password (or the hash). After the interchange is over, Eve (posing as Alice) connects to Bob; when asked for a proof of identity, Eve sends Alice's password (or hash) read from the last session which Bob accepts, thus granting Eve access.

3 Node.js net module

  • From documentation:
    • The net module provides an asynchronous network API for creating stream-based TCP or IPC servers (net.createServer()) and clients (net.createConnection()).
  • HTTP servers are ones that listen for HTTP requests (GET, POST, PUT, etc)
    • Handles data transmitted over connection.
  • TCP/IP servers provide a channel for the communication needs of applications
    • Establishes a connection.
  • The TCP/IP network model consists of four layers (LITA):
    • Link
      • communication over packets
      • Ethernet/WiFi
    • Internet
      • datagrams with sender/recipient information
      • delivers packets from Link over Internet
    • Transport
      • where TCP lives
      • makes sure that data is in order upon delivery, reorders if it got scrambled
      • retransmits dropped datagrams if necessary
    • Application
      • where HTTP lives
      • interprets and communicates high level information
      • handles client requests and delivers them to Transport layer, which uses the Internet layer, which uses the Link layer.
