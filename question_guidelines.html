Asking and Answering Questions
Codecademy Pro Intensives

1 Asking a Question

1.1 Code

  • When asking a question, ensure that you've got a code snippet with you! Having some code to look at helps quite a bit when explaining your issue, and could make it monumentally easier to debug what's going wrong.
  • Slack has a bunch of cool features for dropping in source code.
    • For single line snippets or in-line snippets, you can just use the backtick (`) before and after your line. It's the one that shares a key with the tilde (~). Here's an example:
      • Original: Hey guys! So if I have a line like `alpha = Node("a")`, how is the `next_node` set?
      • Rendered: Hey guys! So if I have a line like alpha = Node("a"), how is the next_node set?
    • For multi-line snippets, you can use three backticks in a row (```) before and after the code block. Here's an example:

      • Original: Hey guys! So I'm working with the Node class here:
        ``` 

        		        class Node:
		            def __init__(self, value, next_node=None):
		       	        self.value = value
		       	        self.next_node = next_node

        ```

      • Rendered: Hey guys! So I'm working with the Node class here: 

                    class Node:
                def __init__(self, value, next_node=None):
                    self.value = value
                    self.next_node = next_node
    • For code that you'd like to get reviewed, you can upload it as a gist and people can submit issues or revisions. If you drop the link in Slack, it'll render the source into a code block for other people to read without having to leave the application.

1.2 Clarity

  • To ensure that everyone can understand your question, make sure to include some examples, and potentially some output. For example, if you're dealing with an error, what you'll want to do is drop the code, the call you made to the code, and the error. Here's a quick example:

    • Hey guys! So I was writing a function to compute the factorial of a number in Python, but I've been getting an error. Any thoughts? Here are the details:

      • SOURCE:
          def factorial(n):
        return n*factorial(n-1)
      • CALL:
          print(factorial(10))
      • OUTPUT:
        RecursionError: maximum recursion depth exceeded

      I think it's something about a missing base case, but I'm not sure.

  • So this example is really great because it defined the context of the problem, the source code that you think is the culprit, the offending call that caused the error, and the error itself.

1.3 Concepts

  • When asking a higher-level, more conceptual question, it still helps a TON to have an example for your peers to reference. Here's an example of how you might as a conceptual question:

    • I've been having trouble understanding how a Python dictionary works. How does it know where to retrieve the values from given just the key? Additionally, how does it ensure access in constant time? This example demonstrates what I'm talking about. 

          # here's a dictionary
    ages = { "Khayyam": 20, "Anna": 25, "Jon": 26, "Tracy": 24 }
    # What happens when we're fetching Jon? How is it O(1) complexity??
    print(ages["Jon"])
  • Even though the question above is purely conceptual, the added example allows your peers answering the question to point to specific places where your concept is applied, and makes it that much easier to understand.

2 Answering a Question

2.1 Structure of an Answer

  • When answering a well-formed question, you want to start by asking your own questions! Make sure that you have full clarity about what they asked, to ensure that you give them the right answer to the right question.
  • Make sure that when you answer a question, you're doing your best to keep it concise. If you go on tangents when providing your answer, the learner can get lost and more confused than when they started!
  • Use bold text or underlines to direct attention to important, key concepts to help them arrive at a solution for their problem.
  • If you're trying to avoid giving them a direct answer to their problem, try using a similar or analogous problem to guide them in the right direction.
  • If you use any fancy terminology, be sure to define it as you go! Hopefully, if there's anything you may have missed, the learner can just ask for clarification, but a lot of the vocabulary can be intimidating, so it's best to break it down for them beforehand.

2.2 The Line Between Enough and Too Much

  • To ensure that whoever you're supporting doesn't get so much help that they're not even thinking about the problem, you've got to tread carefully.
  • As a peer, you've taken on the responsibility of HELPING your fellow learners, without taking away the opportunity to learn.
  • If you think that the concept that they're working through is a CRUCIAL one, make sure you don't give them TOO much when you're helping them.
  • If you're solving their problem for them, you might be doing them more harm than help.
